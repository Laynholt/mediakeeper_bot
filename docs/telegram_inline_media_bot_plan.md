# Технический план проекта: Telegram-бот для инлайн-отправки медиа в чаты

## 1. Краткое описание

Нужно реализовать Telegram-бота, которого можно упоминать в любом чате через `@botname`, чтобы быстро выбирать и отправлять медиа-контент прямо в текущий чат.

Основной сценарий:
- пользователь в любом чате вводит `@botname`;
- Telegram показывает инлайн-результаты от бота;
- пользователь выбирает категорию контента: аудио / изображения / видео;
- при возможности доступен поиск;
- после выбора нужного результата бот подставляет медиа в чат для отправки.

Проект должен быть спроектирован так, чтобы его можно было развивать дальше: добавлять новые источники контента, кэширование, теги, избранное, аналитику, админ-панель.

---

## 2. Цель проекта

Сделать Telegram inline-бота, который:
- работает через inline mode (`@botname query`);
- позволяет находить и вставлять в чат аудио, изображения и видео;
- умеет искать по собственной базе материалов;
- при необходимости поддерживает дальнейшее подключение внешних источников или API;
- имеет понятную архитектуру и готов к расширению.

---

## 3. Основные пользовательские сценарии

### 3.1 Базовый сценарий без текста запроса
Пользователь пишет в чате:
```text
@botname
```

Ожидаемое поведение:
- бот возвращает стартовые inline-результаты;
- в результатах есть быстрые варианты:
  - Аудио
  - Картинки
  - Видео
  - Популярное / Недавнее / Избранное (опционально)

### 3.2 Сценарий с категорией и поиском
Пользователь пишет:
```text
@botname audio rain
```
или
```text
@botname video meme cat
```
или
```text
@botname image sunset
```

Ожидаемое поведение:
- бот определяет категорию;
- выполняет поиск по базе;
- возвращает список релевантных inline-результатов.

### 3.3 Сценарий с произвольным поиском
Пользователь пишет:
```text
@botname rain ambience
```

Ожидаемое поведение:
- бот пытается найти материалы по всем доступным типам;
- результаты могут быть сгруппированы логически или ранжированы по релевантности.

### 3.4 Отправка результата
После выбора результата:
- Telegram вставляет выбранный объект в текущий чат;
- при необходимости добавляется подпись / title / description;
- бот логирует событие выбора (если это включено).

## 4. Ограничения и важные замечания Telegram

Перед реализацией нужно учитывать особенности Telegram Bot API:

1. **Нужен inline mode**
   - бот должен быть создан через BotFather;
   - inline mode должен быть включён.

2. **Inline-результаты ограничены типами Telegram**
   - для аудио можно использовать inline-результаты, совместимые с Bot API;
   - для изображений и видео можно использовать соответствующие inline result types;
   - часть сценариев зависит от того, доступны ли публичные URL для контента или придётся использовать Telegram `file_id` / кэш / проксирование.

3. **Поиск должен быть быстрым**
   - inline-ответы должны формироваться быстро;
   - желательно кэширование результатов поиска.

4. **Не все медиа одинаково удобно отправлять через inline**
   - реализацию стоит строить так, чтобы MVP гарантированно работал хотя бы на части форматов;
   - если какой-то тип контента проблемно отправлять напрямую inline, нужен fallback-подход.

5. **Telegram inline query не даёт полноценный UI с кнопками как в обычном чате**
   - выбор категорий нужно реализовать через текстовый синтаксис запроса, стартовые результаты и подсказки;
   - например: `@botname audio ...`, `@botname image ...`, `@botname video ...`.

## 5. Продуктовое решение

### 5.1 Предлагаемый UX для MVP

Вместо попытки сделать сложный интерфейс внутри inline mode, принять следующий UX:

#### Вариант A — через префиксы категорий
Поддерживать запросы вида:
- `@botname audio <поиск>`
- `@botname image <поиск>`
- `@botname video <поиск>`

Преимущества:
- просто реализовать;
- хорошо сочетается с ограничениями Telegram;
- легко расширять.

#### Вариант B — без префикса, с автоопределением
Поддерживать запросы вида:
- `@botname meme cat`
- `@botname rain`

Бот ищет по всем типам и смешивает результаты.

#### Вариант C — стартовые подсказки
Если пользователь ввёл только `@botname`, бот возвращает help/results:
- `🎵 Аудио` → описание формата запроса
- `🖼 Картинки` → описание формата запроса
- `🎬 Видео` → описание формата запроса
- `🔥 Популярное`

Это можно сделать через inline-результаты-стабсы с текстовыми подсказками.

### 5.2 Рекомендуемое решение
Для первой версии реализовать **комбинацию A + C**:
- основной рабочий сценарий: префиксы `audio`, `image`, `video`;
- при пустом запросе — подсказки и популярные элементы.

## 8. Предлагаемая архитектура

### 8.1 Общая схема

Компоненты:
1. **Telegram Bot Adapter**
   - принимает webhook/polling update;
   - обрабатывает `inline_query`, возможно `chosen_inline_result`, `/start`, админ-команды.

2. **Inline Query Parser**
   - разбирает запрос пользователя;
   - выделяет категорию, текст поиска, фильтры.

3. **Search Service**
   - ищет материалы по базе;
   - возвращает унифицированные результаты.

4. **Media Catalog Service**
   - отвечает за доступ к данным о медиа;
   - абстрагирует БД/файлы/внешние источники.

5. **Result Mapper**
   - преобразует внутреннюю модель медиа в Telegram inline results.

6. **Storage Layer**
   - БД для метаданных;
   - файловое или объектное хранилище для самих медиа;
   - при необходимости таблица кэша и статистики.

7. **Admin / Ingestion Tools**
   - загрузка новых материалов;
   - генерация превью, индексация, заполнение метаданных.

### 8.2 Слои приложения

#### Presentation Layer
- Telegram handlers
- DTO для Telegram событий

#### Application Layer
- use cases:
  - handle_inline_query
  - search_media
  - get_popular_media
  - record_chosen_result
  - import_media

#### Domain Layer
- сущности:
  - MediaItem
  - MediaType
  - Tag
  - SearchQuery
  - SearchResult

#### Infrastructure Layer
- Telegram SDK
- PostgreSQL / SQLite
- Redis cache
- локальное файловое хранилище или S3-compatible storage

## 11. Поиск

### 11.1 Для MVP
Достаточно:
- искать по `title`;
- искать по `description`;
- искать по тегам;
- простое ранжирование по совпадению + usage_count.

### 11.2 Для следующей версии
Добавить:
- полнотекстовый поиск;
- транслитерацию;
- fuzzy matching;
- алиасы;
- языковые нормализации.

### 11.3 Пример логики ранжирования
Score может учитывать:
- точное совпадение в title;
- совпадение по тегам;
- частичное совпадение в description;
- популярность (`usage_count`);
- свежесть (`created_at`).

## 12. Формирование Telegram inline results

Нужно реализовать маппинг по типам:

### 12.1 Audio
Использовать Telegram-compatible inline result для аудио, если доступен подходящий источник.

Нужно проверить в процессе реализации:
- можно ли использовать `telegram_file_id` напрямую;
- нужен ли внешний URL;
- как лучше передавать длительность, performer, title.

### 12.2 Image
Использовать inline result для фото / документ / cached photo в зависимости от способа хранения.

### 12.3 Video
Использовать inline result для видео / cached video / article+preview при fallback.

### 12.4 Fallback-стратегия
Если конкретный тип медиа неудобен для inline-отправки:
- использовать article result с ссылкой / описанием;
- либо хранить Telegram `file_id` и отправлять cached-тип, если Bot API это позволяет;
- либо на этапе ingestion заранее загружать медиа через бота и сохранять `file_id`.

## 15. Рекомендуемый стек

### Язык и фреймворк
Рекомендуемый вариант:
- **TypeScript + grammY**

Почему:
- хорошо подходит для Telegram-ботов;
- удобно структурировать проект;
- хорошая поддержка middleware;
- типизация помогает при росте проекта.

Альтернативы:
- Node.js + Telegraf
- Python + aiogram

Если агенту удобнее Python, допустимо реализовать на **aiogram 3**, но базовая рекомендация — **TypeScript + grammY**.

### Хранилища
Для MVP:
- SQLite или PostgreSQL для метаданных
- локальная файловая система для файлов

Для production:
- PostgreSQL
- Redis для кэша
- S3-compatible storage (опционально)

### Инфраструктура
- Docker
- docker-compose
- `.env`
- webhook или polling

Для простого старта можно использовать polling, но для production желательно webhook.
